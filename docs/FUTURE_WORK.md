# Future work — opportunities for improvement

This is the exhaustive list of improvements that didn't make v1, listed roughly in priority order. The README's [What's next](../README.md#whats-next) section has a curated short version of the most impactful items; this file has the full set with detailed rationale for each.

Each item below is **not implemented in v1**.

---

1. **Close the live-apply loop via a Redis ACL MCP.** The biggest ecosystem gap found during the build: `redis/mcp-redis` is data-plane only — it doesn't expose `ACL CAT`, `ACL LIST`, `ACL SETUSER`, or `ACL WHOAMI`. The plugin reads the server version via `INFO SERVER` but apply is manual. With more time I'd either contribute ACL tools upstream to `redis/mcp-redis`, build a sidecar MCP that wraps `redis-cli` for ACL operations, or extend the Redis Cloud admin MCP (today scoped to subscription/infrastructure) with ACL/user/role endpoints. Closing this enables the safety-gated apply → `ACL GETUSER` verify → impersonation test workflow the agent already has scaffolded.

2. **Enterprise REST API provisioning.** The plugin emits the correct Enterprise rule body but stops short of executing the REST API call to provision it. End-to-end provisioning means generating the JSON payload, authenticating against the cluster manager, POSTing to `/v1/redis_acls`, then attaching the resulting ACL Rule to a Role and assigning the Role to a User. Currently the user pastes the rule body into the admin UI or scripts the REST call themselves.

3. **MCP-driven `MONITOR` verification.** When the agent flags a client-library method whose wire-level command mapping is ambiguous (scripting helpers like `redlock`, locking libraries, transactional pipelines, subcommand-named methods), it could run `MONITOR` against a user-specified test target while the user triggers the code path, capture the actual wire commands, and use those to disambiguate. Closes the loop on "can this be known without reviewing the client library's source." Requires MCP support for `MONITOR` streaming.

4. **Module command category coverage.** RedisJSON (`@json`), RediSearch (`@search`), RedisTimeSeries (`@timeseries`), and the Bloom-filter family (`@bf`, `@cf`, `@cms`, `@topk`, `@tdigest`) are first-class in most Enterprise / Redis Stack deployments but not covered in v1. Our generated `command-category-map.md` pulls from core `redis/redis@8.6.3` only; module commands live in separate repos (`redis/RedisJSON`, `redis/RediSearch`, etc.) and would need their own pulls. The generator script (`scripts/build-category-map.py`) is structured to make this an additive change.

5. **Expand client library coverage + ground the mappings in authoritative sources.** v1 covers `redis-py` (Python), `ioredis` (Node.js), and `go-redis` (Go) only — see `skills/acl-reference/references/client-library-patterns.md`. Languages NOT covered include Java (`jedis`, `lettuce`), .NET (`StackExchange.Redis`), Ruby (`redis-rb`), PHP (`predis`, `phpredis`), and Rust (`redis-rs`, `fred`). Adding them would significantly broaden the plugin's applicability. Additionally, the existing mappings were derived from training data + published API docs, not first-party source-code review. A more rigorous approach: programmatically parse each library's source (or its type stubs / generated API reference) to extract method → command mappings, then verify against `MONITOR` traces (see item #3). That replaces the "method name = command name" convention + hand-curated caveats with derivable, verifiable mappings.

6. **`ACL LOG`-driven denial diagnosis (inverse mode).** Instead of generating an ACL from code, read the audit trail of recent ACL denials on a live Redis, group them by user/command/key, and suggest minimal additions to unblock the application. E.g., *"user `myapp` got denied 47x running `XADD` against `app:events:*` in the last 24h; add `+XADD ~app:events:*` to the rule."* Requires the same MCP ACL surface as item #1.

7. **Runtime-observed ACL generation (`MONITOR` + `ACL LOG` hybrid mode).** v1 infers access patterns by *reading source code* — code-reading has real limits. Dynamic key construction (`r.set(build_key(req), ...)`), conditional command paths, library-internal commands (Sentinel/Cluster client behavior, scripting helpers' wire-level commands), and patterns that only emerge under specific workloads can all be missed, over-estimated, or under-estimated. A runtime-observed mode would capture **what the service actually does** by combining two sources: `MONITOR` against a representative test environment to record the actual command stream while the service runs typical workloads, and `ACL LOG` for any `NOPERM` denials against an existing ACL. The agent then derives the access pattern from observed usage — which commands ran, against which keys/channels/streams, how often — and either generates a fresh rule or compares against the code-inferred rule and surfaces gaps in either direction (*"code says X but runtime shows Y"*). Highest value: a service that's already running can have its ACL tightened to match what it actually does, not what its code *might* do. Generalizes items #3 and #6 into a primary-source workflow. Requires the same MCP `MONITOR` + ACL surface as item #1.

8. **Sub-agent-isolated interactive workflow for production scale.** v1 runs the workflow as an inline skill using `AskUserQuestion`, which is the right call for the demo target (a 50-line sample service). At production scale (a monorepo with thousands of Redis call sites), the discovery output would flood the user's main conversation. The cleaner architecture is to keep the agent fully sub-agent-isolated (context-bounded scan output) and define a structured input/output protocol — agent emits `<NEEDS_USER_INPUT>` blocks, the parent routes them through `AskUserQuestion`, then continues the same sub-agent via `SendMessage` with the user's answers. More complex protocol but main conversation stays clean regardless of codebase size.

9. **ACL selectors (Redis 7+) for finer-grained per-scope permissions.** Today the plugin emits flat rules: `ACL SETUSER alice on >pw ~cache:* ~session:* +SET +GET`. With selectors: `ACL SETUSER alice on >pw +@read (+SET ~cache:*) (+XADD ~stream:logs:*)` — `SET` is granted only on `cache:*`, `XADD` only on `stream:logs:*`. Models real "this command writes only to this key family" intent precisely. v1 deliberately emits flat rules for explainability; selectors are an opt-in tightening.

10. **Multi-client / multi-language analysis in a single pass.** If a codebase uses both `redis-py` and `go-redis` (e.g., a Python web service + a Go worker), v1 asks which client to focus on first and handles one at a time. A future pass could merge the inventories across clients and union the resulting ACL clauses, producing one rule that covers a polyglot service.

11. **End-to-end test coverage for ioredis and go-redis.** v1's `examples/sample-service/` is a `redis-py` fixture and is the only client tested end-to-end (rule generated → applied → service runs cleanly under the locked-down ACL). The `ioredis` and `go-redis` detection patterns are documented in `client-library-patterns.md` from each library's published API but haven't been exercised against real codebases. Adding equivalent `examples/node-sample-service/` and `examples/go-sample-service/` fixtures with their own smoke tests would close the validation gap.

12. **More complex example services.** Related to but distinct from #11: even within a single language, `examples/sample-service/` is a flat ~40-line file exercising strings, pub/sub, and streams under a single key family + single stream + single channel. Real-world Redis-backed services have multi-file structure with key constants split across modules, transactional pipelines (`MULTI`/`EXEC`), Lua scripts via `EVAL`/`EVALSHA`, Redis-backed distributed locks (e.g., `python-redlock` / `node-redlock`), Sentinel or Cluster client mode, and module command usage (RedisJSON, RediSearch, RedisTimeSeries). Adding 2–3 progressively richer fixtures would exercise the agent's handling of ambiguous mappings in realistic settings, surface detection-pattern bugs before customers hit them, and improve demo coverage. The ideal end state is a matrix of (language × complexity) test fixtures spanning trivial → typical → advanced for each supported client library.

13. **Refine `credential-guard` hook heuristics.** The PreToolUse hook blocks file writes containing literal Redis credentials (env-var assignments, URLs with embedded auth, `redis-cli -a <pw>`). It's intentionally aggressive — false positives are possible (a docstring example using `hunter2` will be blocked). The placeholder allow-list (`<changeme>`, `${REDIS_PASS}`, `$REDIS_PASS`, etc.) handles most legitimate cases. Future refinement: entropy scoring on the suspected password (low-entropy strings like `hunter2`/`password123` are likely examples; high-entropy ones are likely real), and an optional warn-mode instead of block.

14. **Auto-fix mode for detected anti-patterns.** Rewrite some client-side anti-patterns in place — `r.keys("user:*")` → `r.scan_iter(match="user:*")`, missing TTL on `SET` → `SET ... ex=N` where the eviction policy doesn't make that safe. Dry-run by default; a PostToolUse hook verifies the rewrite parses. Today the agent's "Style and judgment" section refuses to volunteer anti-pattern warnings because they're context-dependent (eviction policy, cache-vs-primary) — auto-fix would require an explicit user opt-in plus a config interview.

15. **Database-scoped ACLs for multi-tenant Enterprise.** Redis Enterprise lets ACLs be scoped per-database. v1 treats the target as a single ACL surface. A future agent invocation could enumerate the databases in a cluster (via Enterprise REST API), ask which database(s) the rule applies to, and emit per-database bindings.

16. **CI-mode for ACL freshness.** A separate invocation path (`/redis-companion:check`) that takes the current production ACL rule + the latest code, computes whether the rule still covers all of the code's actual usage, and fails the build if the rule is over-broad (grants commands no longer used) or under-broad (a new call site needs grants the rule doesn't include). Turns the plugin from a one-shot dev tool into a continuous-policy enforcer.

17. **Collapse Discovery and Synthesis into a single sub-agent call.** v1's three-phase orchestration dispatches the agent twice — once for Discovery (read code, return findings), once for Synthesis (compose rule from findings + user answers). The split was chosen for separation of concerns and re-runnability (change a user answer, re-run synthesis without re-discovering). In practice, the demo's one-shot usage pattern never exercises re-runnability. Collapsing to one sub-agent would save the wall-time of an extra sub-agent spawn (~10–30s), the token cost of re-delivering the discovery summary verbatim to synthesis (~1–2KB), and the orchestrator's "wait → ask → wait again" double-pause UX. Two enabling changes: (a) the orchestrator does the `INFO SERVER` MCP call inline (without dispatching a sub-agent) before asking Q2, so the version-confirm-or-override pattern still works; (b) the TODO-speculation question becomes a post-hoc nudge in the output (*"I saw a TODO at service.py:40 — re-run me when SUBSCRIBE is wired up"*) rather than a pre-synthesis user choice. Simpler, faster, cheaper.

18. **Multi-version `commands.json` diffing to programmatically derive `version-deltas.md`.** v1's `command-category-map.md` is canonical for one Redis tag (`redis/redis@8.6.3`, pulled from upstream `src/commands/*.json`). The cross-version differences in `version-deltas.md` — category reclassifications, syntax-feature introductions, default-behavior changes — were hand-curated based on training-data knowledge of Redis ACL history, not from primary sources at build time. The major-version shifts (e.g., `EVAL` moving from `@write` to `@scripting` in Redis 7, modules joining `@read`/`@write` in Redis 8) are well-documented and almost certainly correct; specific minor-version cutoffs (e.g., *"selectors landed in 7.2"*) are vulnerable to small errors. The cleaner approach: extend `scripts/build-category-map.py` to pull `commands.json` from multiple tagged Redis releases (6.0, 6.2, 7.0, 7.2, 7.4, 8.0, 8.2, 8.4, 8.6) and diff them programmatically. That would replace the hand-curated category-reclassification section with derivable data. Syntax features (selectors, `%R~`/`%W~` modifiers) aren't in commands.json so would remain hand-curated, but with a much smaller curation surface.
