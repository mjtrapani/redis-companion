## What I'd do with more time

> *(The 2–3 sentence summary the PDF asks for. Detailed roadmap follows below if useful.)*

With more time, my top priority would be **closing the live-apply loop**: the official `redis/mcp-redis` server is data-plane only — `ACL CAT`, `ACL LIST`, `ACL SETUSER`, `ACL GETUSER`, and `AUTH` aren't exposed — so the plugin can detect the server version but can't apply the rule, verify it via `ACL GETUSER`, or impersonation-test the locked-down user. The most direct path is improving `redis/mcp-redis` itself with ACL tools; alternatives are a sidecar MCP wrapping `redis-cli` or extending the Redis Cloud admin MCP (today scoped to subscription/infrastructure) with ACL/user/role endpoints. Beyond that, the highest-value extensions are **Enterprise REST API provisioning** (for cluster-manager-gated deployments), **module command-category coverage** (RedisJSON / RediSearch / RedisTimeSeries — not in core `redis/redis` commands.json), **ACL selectors** for finer per-key-pattern grants in Redis 7+, and **`ACL LOG`-driven denial diagnosis** as an inverse mode that reads recent denials and suggests minimal grants.

## Detailed roadmap

Each item below is **not implemented in v1**. Listed roughly in priority order.

1. **Close the live-apply loop via a Redis ACL MCP.** The biggest ecosystem gap found during the build: `redis/mcp-redis` is data-plane only — it doesn't expose `ACL CAT`, `ACL LIST`, `ACL SETUSER`, or `ACL WHOAMI`. The plugin reads the server version via `INFO SERVER` but apply is manual. With more time I'd either contribute ACL tools upstream to `redis/mcp-redis`, build a sidecar MCP that wraps `redis-cli` for ACL operations, or extend the Redis Cloud admin MCP (today scoped to subscription/infrastructure) with ACL/user/role endpoints. Closing this enables the safety-gated apply → `ACL GETUSER` verify → impersonation test workflow the agent already has scaffolded.

2. **Enterprise REST API provisioning.** The plugin emits the correct Enterprise rule body but stops short of executing the REST API call to provision it. End-to-end provisioning means generating the JSON payload, authenticating against the cluster manager, POSTing to `/v1/redis_acls`, then attaching the resulting ACL Rule to a Role and assigning the Role to a User. Currently the user pastes the rule body into the admin UI or scripts the REST call themselves.

3. **MCP-driven `MONITOR` verification.** When the agent flags a client-library method whose wire-level command mapping is ambiguous (scripting helpers like `redlock`, locking libraries, transactional pipelines, subcommand-named methods), it could run `MONITOR` against a user-specified test target while the user triggers the code path, capture the actual wire commands, and use those to disambiguate. Closes the loop on "can this be known without reviewing the client library's source." Requires MCP support for `MONITOR` streaming.

4. **Module command category coverage.** RedisJSON (`@json`), RediSearch (`@search`), RedisTimeSeries (`@timeseries`), and the Bloom-filter family (`@bf`, `@cf`, `@cms`, `@topk`, `@tdigest`) are first-class in most Enterprise / Redis Stack deployments but not covered in v1. Our generated `command-category-map.md` pulls from core `redis/redis@8.6.3` only; module commands live in separate repos (`redis/RedisJSON`, `redis/RediSearch`, etc.) and would need their own pulls. The generator script (`scripts/build-category-map.py`) is structured to make this an additive change.

5. **`ACL LOG`-driven denial diagnosis (inverse mode).** Instead of generating an ACL from code, read the audit trail of recent ACL denials on a live Redis, group them by user/command/key, and suggest minimal additions to unblock the application. E.g., *"user `myapp` got denied 47x running `XADD` against `app:events:*` in the last 24h; add `+XADD ~app:events:*` to the rule."* Requires the same MCP ACL surface as item #1.

6. **Sub-agent-isolated interactive workflow for production scale.** v1 runs the workflow as an inline skill using `AskUserQuestion`, which is the right call for the demo target (a 50-line sample service). At production scale (a monorepo with thousands of Redis call sites), the discovery output would flood the user's main conversation. The cleaner architecture is to keep the agent fully sub-agent-isolated (context-bounded scan output) and define a structured input/output protocol — agent emits `<NEEDS_USER_INPUT>` blocks, the parent routes them through `AskUserQuestion`, then continues the same sub-agent via `SendMessage` with the user's answers. More complex protocol but main conversation stays clean regardless of codebase size.

7. **ACL selectors (Redis 7+) for finer-grained per-scope permissions.** Today the plugin emits flat rules: `ACL SETUSER alice on >pw ~cache:* ~session:* +SET +GET`. With selectors: `ACL SETUSER alice on >pw +@read (+SET ~cache:*) (+XADD ~stream:logs:*)` — `SET` is granted only on `cache:*`, `XADD` only on `stream:logs:*`. Models real "this command writes only to this key family" intent precisely. v1 deliberately emits flat rules for explainability; selectors are an opt-in tightening.

8. **Multi-client / multi-language analysis in a single pass.** If a codebase uses both `redis-py` and `go-redis` (e.g., a Python web service + a Go worker), v1 asks which client to focus on first and handles one at a time. A future pass could merge the inventories across clients and union the resulting ACL clauses, producing one rule that covers a polyglot service.

9. **End-to-end test coverage for ioredis and go-redis.** v1's `examples/sample-service/` is a `redis-py` fixture and is the only client tested end-to-end (rule generated → applied → service runs cleanly under the locked-down ACL). The `ioredis` and `go-redis` detection patterns are documented in `client-library-patterns.md` from each library's published API but haven't been exercised against real codebases. Adding equivalent `examples/node-sample-service/` and `examples/go-sample-service/` fixtures with their own smoke tests would close the validation gap.

10. **More complex example services.** Related to but distinct from #9: even within a single language, `examples/sample-service/` is a flat ~40-line file exercising strings, pub/sub, and streams under a single key family + single stream + single channel. Real-world Redis-backed services have multi-file structure with key constants split across modules, transactional pipelines (`MULTI`/`EXEC`), Lua scripts via `EVAL`/`EVALSHA`, Redis-backed distributed locks (e.g., `python-redlock` / `node-redlock`), Sentinel or Cluster client mode, and module command usage (RedisJSON, RediSearch, RedisTimeSeries). Adding 2–3 progressively richer fixtures would exercise the agent's handling of ambiguous mappings in realistic settings, surface detection-pattern bugs before customers hit them, and improve demo coverage. The ideal end state is a matrix of (language × complexity) test fixtures spanning trivial → typical → advanced for each supported client library.

11. **Refine `credential-guard` hook heuristics.** The PreToolUse hook blocks file writes containing literal Redis credentials (env-var assignments, URLs with embedded auth, `redis-cli -a <pw>`). It's intentionally aggressive — false positives are possible (a docstring example using `hunter2` will be blocked). The placeholder allow-list (`<changeme>`, `${REDIS_PASS}`, `$REDIS_PASS`, etc.) handles most legitimate cases. Future refinement: entropy scoring on the suspected password (low-entropy strings like `hunter2`/`password123` are likely examples; high-entropy ones are likely real), and an optional warn-mode instead of block.

12. **Auto-fix mode for detected anti-patterns.** Rewrite some client-side anti-patterns in place — `r.keys("user:*")` → `r.scan_iter(match="user:*")`, missing TTL on `SET` → `SET ... ex=N` where the eviction policy doesn't make that safe. Dry-run by default; a PostToolUse hook verifies the rewrite parses. Today the agent's "Style and judgment" section refuses to volunteer anti-pattern warnings because they're context-dependent (eviction policy, cache-vs-primary) — auto-fix would require an explicit user opt-in plus a config interview.

13. **Database-scoped ACLs for multi-tenant Enterprise.** Redis Enterprise lets ACLs be scoped per-database. v1 treats the target as a single ACL surface. A future agent invocation could enumerate the databases in a cluster (via Enterprise REST API), ask which database(s) the rule applies to, and emit per-database bindings.

14. **CI-mode for ACL freshness.** A separate invocation path (`/redis-companion:check`) that takes the current production ACL rule + the latest code, computes whether the rule still covers all of the code's actual usage, and fails the build if the rule is over-broad (grants commands no longer used) or under-broad (a new call site needs grants the rule doesn't include). Turns the plugin from a one-shot dev tool into a continuous-policy enforcer.
